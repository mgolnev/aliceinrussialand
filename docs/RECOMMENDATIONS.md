# Рекомендации постов (логика)

Описание того, **как сервер отбирает посты** для блоков «дальше читайте» на странице публикации и «ещё вдохновения» в конце ленты категории. Поведение UI здесь не разбирается — только источники данных и правила ранжирования.

## Общие понятия

### Качество поста в ленте

Везде, где нужно упорядочить посты «как в основной ленте», используется один и тот же приоритет:

1. **Закреплённые** (`pinned`) выше остальных.
2. Затем **дата публикации** (`publishedAt`), новее выше.
3. При равенстве — стабильная сортировка по **`id`** (убывание).

То же сравнение двух постов вынесено в `compareCarouselQuality` в [`src/lib/rec-diversify.ts`](../src/lib/rec-diversify.ts).

### Разнообразие по категориям (round-robin)

Функция `diversifyByCategoryRoundRobin` ([`src/lib/rec-diversify.ts`](../src/lib/rec-diversify.ts)) получает список постов, который **уже отсортирован по качеству** (см. выше). Посты без категории считаются одной группой с ключом `__uncat__`.

Алгоритм:

1. Посты раскладываются по **корзинам** по `categoryId`, внутри каждой корзины порядок совпадает с исходным (то есть «лучший» в теме — первый в корзине).
2. Порядок обхода тем фиксируется по **первому появлению** темы в исходном списке (раньше встретилась в отсортированной ленте — раньше участвует в раундах).
3. Пока не набрано нужное число элементов, на каждом **раунде** из каждой темы берётся **не больше одного** поста (снимается с головы корзины). Так в выборку попадают сначала лучшие представители разных рубрик, потом вторые по качеству и т.д.

Это уменьшает ситуацию, когда весь блок заполняется материалами одной категории, хотя в базе есть сильные посты и в других темах.

### Соседние категории

**Соседи** — это предыдущая и следующая категория в порядке **`sortOrder`** (как в админке / в списке тем ленты). Для страницы поста порядок берётся из `postCategory` через Prisma ([`listCategoryIdsByFeedOrder` в `posts-query`](../src/lib/posts-query.ts)). Для конца ленты категории — из [`listFeedCategories`](../src/lib/feed-server.ts) (тот же смысл порядка, плюс кэш).

## 1. Карусель после поста (`getPostCarouselPeers`)

**Файл:** [`src/lib/posts-query.ts`](../src/lib/posts-query.ts)  
**Кэш на запрос страницы:** `getPostCarouselPeersCached` (React `cache`).

**Параметры по умолчанию:** `categoryFirst = 5`, `totalLimit = 16` (можно переопределить в `opts`).

### Шаг A — та же категория

Если у текущего поста есть `categoryId`:

- выбираются до **5** опубликованных постов **в этой категории**;
- текущий пост **исключается**;
- сортировка — по качеству ленты (закреп → дата → id).

Если категории нет, этот блок пустой.

### Шаг B — хвост до `totalLimit`

Оставшиеся слоты заполняются отдельно.

1. **Соседние рубрики** определяются по позиции текущей категории в списке id категорий по `sortOrder`. Если категории у поста нет или она не найдена в этом списке, соседей нет.
2. Запрос **«соседи»:** до **40** постов из соседних категорий, не входящих в уже выбранные id (текущий пост + посты из шага A), сортировка по качеству ленты.
3. Запрос **«остальные»:** до **40** постов, которые:
   - не входят в `excludeIds` после соседей (то есть исключаются текущий пост, шаг A и все id из выборки соседей);
   - по категории: либо **без категории**, либо категория **не** текущая и **не** соседняя (чтобы не дублировать соседний пул).

   Если запретов по категориям нет (крайний случай), берутся любые подходящие по `id`.

4. Списки **склеиваются** в порядке `[соседи, остальные]`. Так в общем приоритете сначала идут кандидаты из **тематически близких** рубрик, затем — из остальных.
5. Из склеенного списка функция **`diversifyByCategoryRoundRobin`** набирает ровно **`remaining`** постов.

Итоговый порядок для UI: **сначала блок из той же категории, затем хвост** с приоритетом соседей и размазанным по категориям составом.

## 2. Блок в конце ленты категории (`getCategoryFeedExplore`)

**Файл:** [`src/lib/category-feed-explore.ts`](../src/lib/category-feed-explore.ts)  
**Данные:** `CategoryFeedExplorePayload` — главный пост (`featured`), карусель (`more`), чипы других тем (`topics`).

В **production build** с флагом `isNextProductionBuild()` функция возвращает `null` (блок не строится) — см. код.

### Главный пост (`featured`)

1. Если есть **соседние** категории (по slug текущей ленты в `listFeedCategories`):
   - для **каждой** соседней категории отдельно берётся **один** лучший пост (`findFirst` с тем же порядком: закреп → дата → id);
   - из этих «топов соседей» выбирается **один** лучший по `compareCarouselQuality`.

   Так главный слот не прилипает к одной соседней рубрике только потому, что в ней всегда чуть новее материал: сравниваются **равные по правилам** кандидаты из разных соседей.

2. Если соседей нет или в них нет опубликованных постов — **fallback:** один лучший пост среди всех категорий **кроме текущей** (`categoryId not current`).

### Карусель (`more`)

- Исключается id **featured** (если он есть).
- Загружается до **28** кандидатов: опубликованные, **не** в текущей категории, сортировка по качеству ленты.
- Из этого списка **`diversifyByCategoryRoundRobin`** отбирает **6** постов — снова с упором на **разные категории** в первую очередь.

### Чипы тем (`topics`)

Первые **5** категорий из полного списка ленты **кроме текущей** (порядок как в `listFeedCategories`). Отдельного алгоритма «рекомендаций» здесь нет — это навигация по темам.

## Ограничения и краевые случаи

- **Нет тегов и ML:** только категория поста, порядок категорий и поля `pinned` / `publishedAt`.
- **Мало контента:** round-robin и лимиты запросов могут дать короче списка, чем лимит — это нормально.
- **Пост без категории:** блок «та же категория» пуст; хвост строится из общих правил (без привязки к соседям по индексу текущей темы).

При изменении чисел **5**, **16**, **40**, **28**, **6** имеет смысл обновить этот документ и комментарии в коде.
